# 가장 잘했다고 생각하는 부분

이번 프로젝트를 진행하면서 가장 신경 쓴 부분은 객체지향 설계와 입력 검증 로직의 효율화였습니다. 
처음에는 각 클래스마다 비슷한 검증 코드가 들어가서 중복이 많았는데 이를 해결하기 위해 Transaction이라는
부모 클래스를 만들고 금액, 카테고리, 날짜, 설명 등의 공통 검증 로직을 한데 모았습니다. 

덕분에 하위 클래스인 Income과 Expense에서는 super().__init__()을 통해 공통 로직을 
자연스럽게 재사용할 수 있었고 각자의 고유 카테고리나 금액 부호 처리 그리고 세부 기능에만 
집중할 수 있는 깔끔한 구조를 만들 수 있었습니다. 배포 가능한 수준의 패키지를목표로 삼았기에 
패키지 빌드 설정과 테스트, 스타일 검사(PEP 8), 커버리지 관리 그리고 기본적인 
CI 워크플로우 예시까지 구성했습니다. 외부에서 호출하는 공개 클래스와 메서드에는 docstring을
꼼꼼히 작성했고 핵심 메서드에는 닥테스트(doctest)가 가능하도록 >>> 형태의 사용 예시를 
직접 추가하여 코드의 신뢰성을 높이고자 노력했습니다.

# 아쉬운 부분

현재 버전은 메모리 안에서 거래를 관리하므로 프로그램을 종료하면 데이터가 저장되지 않습니다. 
저장 기능은 포함하지 않았지만, 실제 사용성을 생각하면 CSV 또는 JSON 저장과
불러오기 기능이 있으면 실제로 사용하는데 있어서 더 좋을 것 같습니다. 

또한 카테고리는 안정적인 검증을 위해 미리 정해둔 값만 허용하도록 제작했습니다. 
정해둔 값만 허용하니 오타를 줄이는 장점이 있지만, 사용자가 직접 카테고리를 추가하고 싶은
경우에는 제한적으로 느껴질 수 있다고 생각합니다.

## 1. README.md 필수 구성 내용
안내서에서 **6가지 항목을 필수**로 지정하고 있으며, 처음 보는 사람도 이 파일만 보고 패키지를 실행할 수 있을 만큼 구체적이어야 합니다.
**(1) 프로젝트 개요**: 패키지가 무엇을 하는 도구인지 (예: 가계부 및 예산 추적기) 명확하게 설명.
**(2) 설치 방법**: 가상환경에서 `pip install .` 명령을 통해 패키지를 설치하는 방법 기술.
**(3) 빠른 시작 (Quick Start) 코드 예시**: 사용자가 코드를 그대로 복사해서 붙여넣었을 때 바로 동작하는 주요 클래스 및 메서드 사용 예시 코드 블록.
**(4) 주요 기능 설명**: 패키지가 제공하는 핵심 기능들에 대한 상세 안내.
**(5) 테스트 실행 방법**: `pytest`를 활용하여 테스트를 구동하는 명령어 안내.
**(6) 작성자 정보**: 본인의 정보(학번, 이름 등) 기재. 
**➕ 필수 포함**: 결과 요약 파일 및 자기 평가서와 함께 **GitHub 저장소 URL**을 반드시 이 README 내에 포함하거나 ZIP 업로드 시 확인 가능해야 합니다.

## 2. results.md (실행 결과 요약) 필수 구성 내용
패키지의 품질 검증 도구들을 실행하고, 그 결과를 증명하는 **스크린샷 또는 텍스트 출력 캡처**가 반드시 포함되어야 합니다.
**pycodestyle (또는 flake8) 실행 결과**: 정적 분석 도구를 돌렸을 때 경고(Warning)가 없거나 최소화된 상태의 터미널 출력 결과.
**pytest 실행 결과**: 프로젝트 루트에서 `pytest`를 실행하여 작성한 모든 테스트 케이스가 성공(`failed: 0`)했음을 보여주는 터미널 출력 결과.
* pip install . 실행 결과: 새 가상환경에서 패키지가 에러나 사소한 경고 없이 한 번에 정상 설치되는 과정을 보여주는 터미널 출력 결과.

학업 윤리 및 AI 도구 사용 정책과 연계되는 중요한 서류입니다. 다음 세 가지 항목을 **각각 1~2문단으로 서술**해야 합니다.
**내가 가장 잘했다고 생각하는 부분**: 객체지향 설계(DRY 원칙, super() 활용), 문서화, 테스트 케이스 작성 등 프로젝트 중 가장 자신 있는 부분 강조.
**아쉬운 부분**: 일정 관리, 예외 처리 범위, 혹은 미처 구현하지 못한 기능에 대한 솔직한 반성.
**다음에 개선하고 싶은 점**: 향후 코드를 어떻게 리팩토링하거나 고도화하고 싶은지 발전 방향 기술.

## 🚨 작성 및 제출 시 핵심 유의사항 (감점 방지 체크리스트)

### ① 코드 스타일 및 Docstring 연계 주의 (PEP 8)
* 앞서 수정하셨던 것처럼 `pycodestyle`이나 `flake8`을 실행했을 때 경고가 5건 이하(가장 좋은 것은 0건)여야 합니다. Docstring에 한글 설명을 넣으면서 한 줄이 79자를 넘지 않도록 자연스럽게 줄바꿈을 해주세요.
* 모든 공개(Public) 클래스, 메서드, 함수에는 반드시 docstring이 있어야 하며, 설명뿐만 아니라 최소 1개 이상의 사용 예시(`>>>` 형태)를 포함하여 doctest가 가능하게 해야 '우수' 등급을 받습니다.

### ② 테스트 케이스 구성 (pytest)
* 테스트 함수명은 `test_`로 시작해야 합니다.
**최소 8개 이상의 테스트 케이스**를 작성하되, 정상 동작 5건 외에 **'빈 입력', '잘못된 타입', 'None', '경계값' 등의 엣지 케이스(Edge Case)를 최소 3건 이상** 포함해야 감점이 없습니다.

### ③ Git 커밋 로그 및 관리
* 로컬 저장소에 **최소 7개 이상의 의미 있는 커밋**이 찍혀있어야 합니다.
* 한 번에 코드를 다 짜고 몰아서 커밋을 남기면 미흡 처리를 받습니다. 또한 커밋 메시지에 `"update"`, `"fix"`, `"수정"`처럼 모호하게 적지 말고, 명확한 명령형 동사로 구체적인 변경 내용을 적으세요 (예: `"Add Expense subclass and implement budget check"`).
`.gitignore` 파일을 올바르게 설정하여 `__pycache__/`, `.venv/`, `.pytest_cache/` 같은 불필요한 캐시나 가상환경 파일이 Git에 추적되지 않도록 하세요.

### ④ ZIP 파일 최종 제출 규격
* 최종 제출 ZIP 파일 안에는 **`README.md`, 실행결과 요약 파일, 자기 평가서가 모두 포함**되어야 하며, 불필요한 `__pycache__`나 `.venv` 폴더는 제외해야 합니다.
* 파일명 규격: **`학번_이름_finalproject.zip`** (예: `20241234_홍길동_finalproject.zip`).
* 마감 시간 엄수: 24시간 지각 시 10% 감점, 48시간 이후에는 0점 처리되므로 시간을 넉넉히 두고 GitHub 푸시 및 업로드를 확인하세요.'
